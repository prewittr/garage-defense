# [RPM-008] Input Latency Gate (<12ms p95)

**Feature:** [FEAT-001](FEAT-001-kinetic-door.md) (quality gate)
**Owner:** **Kendra Brooks** (lead) + **Jasmine Whitfield** (support)
**Size:** S
**Risk:** 🟡 — device-farm probe is Sprint 2+; Sprint 1 validates on Editor + desktop profiler only
**Status:** **Ready**

## User Story
**As the** tech lead
**I want** drag-input-to-visual latency instrumented with a p95 gate wired into CI
**So that** feel-killing latency regressions fail the build the moment they land — not the day we demo to Owner

## Acceptance Criteria
- [ ] **Given** a drag gesture in `Rpm.Input`, **when** the frame renders the scrap sprite following the finger/cursor, **then** the delta `t_render − t_input` is captured in milliseconds.
- [ ] **Given** a dedicated test scene `Scenes/LatencyProbe.unity`, **when** a scripted drag performs 100 samples across 5 seconds, **then** p50 and p95 latency are written to `Builds/latency-report.json`.
- [ ] **Given** the report file, **when** the CI `latency-gate` job reads it, **then** p95 ≤ 12ms passes and p95 > 12ms fails the build.
- [ ] **Given** the gate runs in CI, **when** any result lands, **then** p50 and p95 are echoed in the PR status-check description (not just pass/fail).
- [ ] **Given** a Sprint 1 baseline, **when** RPM-001 merges, **then** the reported p95 on ubuntu-latest desktop profiler is recorded in a `latency-baseline.md` in `docs/qa/` for future regression comparison.

## Notes
- Ubuntu-latest runners are **desktop-class**, not mobile-class. The <12ms gate is meaningful on CI desktop as an early-warning; real device-class latency (iPhone 12 / Pixel 6) must be measured Sprint 2 onward via physical device or cloud device farm (e.g. BrowserStack, Firebase Test Lab).
- If Sprint 1 reveals ubuntu-latest runners can't reliably hit sub-12ms even with clean code (runner jitter), Kendra files a Redline Escalation card to Marissa and we re-spec the gate to "absolute wall-clock p95 < 12ms" vs. a "frame-count" measurement.
